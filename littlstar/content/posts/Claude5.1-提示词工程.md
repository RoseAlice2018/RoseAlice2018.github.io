### 1. Claude 的 Prompt 是如何设计的？

Claude 的 Prompt 设计采用了**"静态和动态分离"**的原则，并在业界最佳范式中推荐使用**"三段式汉堡包结构"**。具体设计如下：

*   **静态块（Static Prompt）：**
    *   **内容：** 包含尽量稳定的信息，例如系统规范、工具 Schema、语气风格、角色设定、全局核心规则和不变的背景知识。
    *   **位置：** 放置在最前面（头部）。有时也会在尾部设置一个极简的静态块（用于再次强调关键的输出格式或最后一条限制）。
*   **动态块（Dynamic Prompt）：**
    *   **内容：** 包含高频变化的信息，如当前会话（Current Conversation）、记忆（Memory）、环境信息（如当前时间、日期等）、Claude.md MCP 插件信息、Token 预算提示，以及检索到的外部参考资料。
    *   **位置：** 放置在静态头部之后，作为中部或独立的消息节点（紧贴用户真实输入之前）。

**最佳业界范式（三段式汉堡包结构）：**
1.  **静态头部（最大化 Cache）：** 角色设定、全局核心规则、不变的背景知识。
2.  **动态中部：** 检索到的参考资料、用户当前对话上下文。
3.  **静态尾部（通常不走 Cache）：** 极简，再次强调最关键的输出格式或限制，利用近因效应确保格式正确。

### 2. 为什么先是静态块后是动态块？对缓存与一致性有什么影响？

将静态块放在动态块之前，是出于性能（缓存）和模型推理质量（一致性）的双重考量：

#### 对"缓存"（性能）的影响：极大提高命中率，降低成本和延迟
*   **前缀匹配机制：** 主流的 LLM 推理加速技术（如 vLLM 的 KV Cache、Anthropic/OpenAI 的 Prompt Caching）大多基于前缀匹配，系统从 Prompt 的第一个 Token 开始计算哈希。
*   **最大化复用：** 静态块对所有用户都是一样的。把它放在最前面，系统可以完美命中并复用这部分长文本的 KV Cache。
*   **避免"缓存灾难"：** 如果将高频变化的"环境信息"（动态内容）混入前面的静态块，会导致每次请求的前缀 Token 发生微小变化。这会引起：
    1.  前缀缓存彻底失效（退化为零命中）。
    2.  KV Cache 频繁换页，严重浪费显存，引发系统抖动。
    3.  首字延迟（TTFT）飙升，原本能跳过的计算现在必须重新执行。
    4.  算力成本翻倍（命中缓存的 Token 价格远低于未命中的 Token）。

#### 对"一致性"（推理质量）的影响：防止基调漂移和提示词注入
LLM 依赖自注意力机制，具有显著的"首因效应"（Primacy Bias）和"近因效应"（Recency Bias）。
*   **锚定角色与基调（利用首因效应）：** 模型在开头先读取静态规则（如"你是一个严谨的助手"），其内部隐式状态会立刻被锚定在正确的概率分布上。如果先读杂乱的动态上下文（先动后静），模型的注意力会被带偏，导致输出基调发生漂移，一致性变差。
*   **防止指令被污染（Prompt Injection）：** 动态块通常包含不可控的外部数据（如联网搜索）。如果动态内容放在最前面，恶意数据可能伪装成系统指令。先放静态规则能建立"免疫护栏"，降低模型受诱导产生幻觉或越狱的风险。
*   **避免约束条件被遗忘：** 如果把所有静态的边界和规则都放在庞大的动态长文本之后，模型很难将这些规则有效地反向应用到刚刚读完的复杂上下文中（即"迷失在中间"现象）。

总结来说，**先静后动**在工程上榨取了最大的缓存价值，在算法上锚定了模型的核心身份和安全边界，是确保性能与一致性的最优解。
